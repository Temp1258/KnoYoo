## 产品初衷

每个人每天都在消费大量信息——网页文章、技术博客、YouTube/B站视频、电子书、PDF 报告——但绝大多数内容看完就散落了。书签收藏夹变成坟场，浏览历史被冲走，笔记散落在五个不同的 App 里。

**信息被消费了，但知识没有被积累。**

这不是用户的错。现有工具要么只解决"存"的问题（Pocket、书签），要么需要大量手动整理才能发挥价值（Notion、Obsidian），要么把你的数据锁在别人的服务器上（Evernote、各种云端服务）。

KnoYoo 要解决的不是"怎么记笔记"，而是一个更根本的问题：

> **如何让被动消费的信息，以最低成本转化为可复用的个人知识资产，并且这些资产 100% 属于你自己。**

理想状态是：你只需要像平时一样浏览、阅读、看视频，KnoYoo 在后台帮你把这些消费过的信息变成永远不会丢失、随时可以找回的结构化知识。你不需要改变任何习惯，不需要学习任何整理方法论——存下来的那一刻，AI 已经替你完成了整理。

## 核心定位

KnoYoo 是一个**本地优先的 AI 私人智库**。

- **本地优先**：所有数据 100% 存储在用户本地 SQLite 数据库，零云端上传，零遥测，用户对自己的数据拥有完全控制权。
- **AI 驱动**：利用大语言模型自动完成内容清洗、摘要提炼、标签分类、语义搜索、知识对话，将"存"之后的整理工作降到零。
- **私人智库**：不是笔记工具，不是书签管理器，不是稍后读应用。它是一个接收多种来源信息、自动结构化、支持智能检索和 AI 对话的个人知识引擎。

这个定位建立在四大支柱之上。

## 四大支柱

### 支柱一：无痛输入 + 自动结构化（存）

> **使命：把信息变成知识。用户只需要做"存"这一个动作，剩下的交给 AI。**

#### 为什么这是核心

大多数知识管理工具死在同一个地方：**输入成本太高**。用户需要复制粘贴、手动分类、写标签、写摘要——这些动作在收藏三五条时可以忍受，但当你每天消费几十条内容时，整理的负担会迅速压垮使用意愿。

KnoYoo 的策略是：**把输入成本压到最低（一键/拖入），把整理成本压到零（AI 自动完成）。** 用户唯一需要做的决策是"这个内容值不值得存"，其他所有事情由系统接管。

#### 无痛输入：多源接入

不管内容来自哪里、什么格式，存进来的动作都是一步到位的：

| 输入源 | 动作 | 实现方式 |
|---|---|---|
| 网页文章 | 浏览器扩展一键点击 | Content Script 提取 DOM 文本 + 背景 Worker 推送至桌面端 |
| YouTube 视频 | 收藏链接 | 解析 `ytInitialPlayerResponse`，优先抓 publisher 字幕，没有则用 ASR 自动字幕 |
| B站视频 | 收藏链接 | 调用 `x/web-interface/view` 公开 API，支持 BV 号、短链、追踪链接自动清洗 |
| EPUB 电子书 | 拖入桌面端窗口 | 解析 EPUB 元数据 + 封面提取 + AI 通读前万字 |
| PDF 文件 | 拖入桌面端窗口 | 双路抽取：`pdf-extract`（主路径，兼容 ToUnicode CMap）+ `lopdf`（兜底逐页） |
| 浏览器书签 | 批量导入 | 解析 Netscape 书签格式（Chrome/Firefox/Edge 通用） |

**离线兜底**：浏览器扩展内置离线队列（最多 100 条），桌面端未运行时自动排队，上线后批量同步。队列溢出时主动通知用户。

**去重保护**：`web_clips(url)` 唯一索引防止重复收藏；书籍通过 `file_hash` 去重（软删除的文件不阻挡重新导入）。

**自动弹窗**：浏览器扩展在每个新页面自动弹出 Shadow DOM 浮窗，8 秒后自动消失，提供一键存入入口。用户可在设置中关闭此行为。

#### 自动结构化：三阶段 AI 管道

内容存入后，AI 自动完成三步加工，**这是 KnoYoo 与"高级书签管理器"的根本分水岭**：

**Stage 1 — 原文保留（Raw）**

同时保留两路原文：浏览器渲染后的可见文本（Content Script 提取）+ 服务端 SSRF-safe 抓取的 HTML 正文。两路互相兜底，确保 SPA 站点（React 类应用）不会只留下一个标题。如果客户端提取的内容过短（< 100 字符），自动触发服务端补抓。

**Stage 2 — AI 清洗（Readable）**

将原始内容交给 AI 清洗为可读的 Markdown 正文。关键约束：

- **清洗，不压缩**：AI 被明确要求保留原文信息，严禁概括
- **Size Guard**：清洗结果不能压缩原文超过 66%，否则拒绝覆盖，回退到原始内容
- **原文永远保留**：清洗后的可读版写入 `content` 字段，原始文本保留在 `raw_content`，UI 支持「可读版 / 原始」一键切换

**Stage 3 — 摘要 + 标签（Summary + Tags）**

基于清洗后的正文，AI 生成：

- 精炼中文摘要
- 3-5 个关键词标签（复用已有标签体系保持分类一致）
- 来源分类

所有 AI 任务在后台线程池运行（最多 8 个并发），不阻塞 UI。失败自动重试一次，支持手动重跑。

#### 书籍的自动结构化

书籍入库后，AI 通读正文前约 12,000 字符（封面页 + 版权页 + 目录 + 前言 + 第一章），自动填充：

- 真实标题（过滤 "Microsoft Word - xxx.doc" 这类残留）
- 作者、出版社、出版年份
- 2-4 句中文简介
- 3-5 个关键词标签

AI 状态通过 `ai_status` 字段追踪（`pending → ok / failed`），UI 实时显示处理状态。纯扫描版 PDF 无法抽取文字时，自动以文件名作为占位标题，显示警告角标和重试按钮。

#### 视频的自动结构化

视频导入后，系统按八段式管道处理：

```
元数据获取 → 字幕探测 → 音频下载 → [分片] → ASR 转写 → AI 清洗 → 摘要标签 → [双语翻译]
  0-5%       5-10%      10-40%    40-45%   45-80%    80-95%    95-98%     98-100%
```

字幕优先策略：有 publisher 字幕用字幕（质量最高）→ 有自动字幕用自动字幕 → 都没有时下载音频走云端 ASR（OpenAI Whisper / Deepgram / SiliconFlow 三选一）。音频超过供应商单次上传上限时 ffmpeg 自动按时长切片串行转写。完成后复用 Stage 2-3 管道进行清洗和标签生成；若源语言不是简体中文，最后追加一次 AI 翻译生成双语字幕。

前端通过 Tauri 事件 `transcribe://progress` 实时接收进度，展示分阶段进度条和当前动作文案。

### 支柱二：智能检索 + AI 对话（用）

> **使命：在需要的时候精准找回，并能与自己的知识库对话。**

#### 为什么这是核心

知识库如果只进不出，就是数字仓鼠症。用户存了 1000 条内容，"然后呢？"——这个问题的答案决定了产品是否有真正的价值。

KnoYoo 的价值在"存"之后才真正兑现：**当用户在写方案、做研究、准备面试时，能在几秒内从自己的智库里找到三个月前看过的那篇文章、那个视频片段、那本书里的某个概念。**

#### 三层检索体系

**第一层：全文搜索**

- 基于 SQLite FTS5 全文索引，覆盖标题、正文、标签、摘要
- 毫秒级响应，即打即搜
- 搜索历史持久化，支持逐条删除
- 已实现并作为默认搜索方式

**第二层：AI 语义搜索**

- 当全文搜索无结果时自动兜底
- 用户可以用自然语言描述来找内容："那篇讲分布式锁最佳实践的文章"、"之前看过一个用 Rust 写 CLI 的教程"
- AI 根据语义理解匹配智库中的相关内容
- 已实现，通过 `ai_fuzzy_search_clips` 命令

**第三层：AI 知识对话**

- 用户直接向自己的知识库提问
- 系统将用户的收藏内容、集合、笔记作为上下文注入对话
- AI 回答**附带引用来源**，用户可以直接跳转到原始内容
- 对话会话持久化，可以跨时间继续追问
- 已实现，支持 `ai_chat`（通用对话）和 `ai_chat_with_context`（带智库上下文的对话）

#### 多维度筛选

除了搜索，用户还可以通过多个维度缩小范围：

| 维度 | 说明 |
|---|---|
| 标签筛选 | 点击任意标签快速过滤 |
| 域名筛选 | 按来源网站过滤（比如只看来自 GitHub 的内容） |
| 时间范围 | 按收藏时间筛选 |
| 星标 | 只看收藏了星标的重要内容 |
| 已读/未读 | 区分已阅读和待阅读的内容 |
| 集合 | 主题 + 颜色 + 描述，手动归类的知识分组 |

#### 相关推荐

打开一条收藏时，系统自动推荐相关内容（通过 `find_similar_clips` 提取关键词匹配）。这种被动发现机制让用户在阅读时自然地建立知识间的联系。

#### 规划中的增强

| 功能 | 描述 | 状态 |
|---|---|---|
| 全局快捷键搜索 | `Cmd+K` 随时唤起搜索，不需要先打开主窗口 | 规划中 |
| 智能集合 | 基于规则自动归类内容（如"所有包含 Rust 标签的内容"） | 规划中，`filter_rule` 字段已预留 |
| 跨内容类型搜索 | 同时搜索剪藏、书籍、笔记，统一排序 | 规划中 |

### 支柱三：知识生长可视化（感）

> **使命：让无形的知识积累变成有形的成长感知，用正反馈驱动持续使用。**

#### 为什么这是核心

工具类产品最大的敌人是"用着用着就忘了"。用户第一周热情高涨，第二周偶尔想起来用一下，第三周就回到了老习惯。

打破这个循环的关键不是更多功能，而是**让用户感受到积累的价值**。当你能清楚地看到"我这个月在 AI 领域收藏了 47 篇文章，形成了 3 个主题集合，比上个月多了 60%"——这种成长的可视化本身就是最强的留存引擎。

#### 已实现的可视化能力

**发现页（DiscoverPage）**

发现页是 KnoYoo 的"知识仪表盘"，目前包含以下模块：

- **标签词云**：按使用频率缩放标签大小，直观展示知识分布的形状——你在哪些领域积累最多，一眼可见
- **来源 Top 5**：条形图展示内容来源最多的 5 个域名，揭示你的信息获取习惯
- **28 天趋势**：柱状迷你图展示过去 28 天每天的收藏数量，可视化学习节奏和连续性
- **AI 周报**：每周自动生成，AI 分析本周收藏的所有内容，输出学习总结和洞察
- **遗忘内容提醒**：随机浮现 5 条超过 30 天未查看的内容，主动对抗知识沉底

**书架系统**

- 按阅读状态分组：想读 / 正在读 / 已读 / 弃读
- 阅读进度百分比 + 评分系统
- 封面展示（EPUB 自动提取 / PDF 生成色块封面）

#### 规划中的深化方向

当前的可视化更偏"统计仪表盘"——展示数量和分布。要真正实现"成长正反馈"，需要从**统计**升级到**叙事**：

**知识时间轴**

不只是"过去 28 天你存了多少条"，而是一条可以回溯的时间轴，展示知识积累的**故事**：

- "你在 2024 年 3 月开始关注 Rust，4 月扩展到系统编程，5 月开始实践 CLI 工具开发"
- 标签的出现时间和增长轨迹，让用户看到自己兴趣的演变
- 按月/季度的知识积累对比

**里程碑与仪式感**

给日积月累赋予仪式感，在关键节点给出正向激励：

- 数量里程碑："你的智库突破 500 条"、"第 100 本书入库"
- 连续性里程碑："连续 30 天有新输入"、"本月每天都在学习"
- 深度里程碑："你在『机器学习』主题下已经积累了 50 篇内容"
- 阅读里程碑："你已经读完第 10 本书"

**知识网络图**

从扁平的标签列表升级到可交互的知识网络：

- 基于标签共现关系构建节点-边图
- 展示内容之间的关联："这 8 篇文章共享『分布式系统』和『一致性』标签，形成了一个知识簇"
- 让用户直观地看到自己的知识不是碎片，而是有结构的网络
- 点击节点可以跳转到对应的内容列表

**主动复习引擎**

从被动的"遗忘提醒"升级为主动的复习推荐：

- 基于间隔重复（Spaced Repetition）的原理，在最佳时间点推荐回顾
- "你 3 个月前深入研究过 React Server Components，相关领域最近有 2 条新收藏，要不要回顾？"
- 将"遗忘"转化为"复习机会"，赋予积极含义

**学习目标追踪**

允许用户设置学习方向，系统自动追踪进展：

- "我想深入学习 Kubernetes"→ 系统追踪相关收藏数量、阅读进度、知识覆盖度
- 进度可视化：从"入门"到"系统性了解"的阶段展示

### 支柱四：100% 本地 + 用户拥有数据（信）

> **使命：用户愿意把全部知识托付给一个系统的前提是——这个系统不会背叛我。**

#### 为什么这是核心

支柱四不是一个功能，而是一个**信任基础**。如果用户不信任系统，前三个支柱全部失效——没有人会把敏感的行业报告、私人笔记、职业规划存入一个可能泄露数据的工具。

在 AI 时代，这个问题变得尤其尖锐：用户想要 AI 的智能能力（自动整理、语义搜索、知识对话），但同时担忧 AI 服务商会访问自己的数据。KnoYoo 提供了一个第三选择：**享受 AI 的智能能力，但数据永远不离开你的电脑。**

#### 已实现的隐私保障

**数据层**

| 措施 | 说明 |
|---|---|
| 本地 SQLite | 所有剪藏、书籍、笔记、对话历史 100% 存储在本地文件系统 |
| 零云端上传 | 不存在任何将用户数据上传至 KnoYoo 服务器的代码路径 |
| 零遥测 | 不收集任何使用数据，不使用埋点、分析服务 |
| 路径可见 | 设置页明确显示数据库文件路径和占用空间 |
| 完全导出 | 支持 Markdown 导出和 SQLite Backup API 完整数据库备份/恢复 |

**网络层**

| 措施 | 说明 |
|---|---|
| 本地监听 | HTTP 服务仅绑定 `127.0.0.1:19836`，其他设备无法访问 |
| SSRF 防护 | URL 抓取阻断 localhost / 私有 IP / link-local / `.internal` / `.local` |
| DNS Rebinding 防御 | 自定义 resolver 在连接时二次验证 DNS 结果 |
| 严格 CSP | `connect-src` 白名单限定 AI 供应商域名 |
| 输入长度限制 | URL 4KB / 内容 500KB / 聊天 5MB / 书籍 500MB |

**认证层**

| 措施 | 说明 |
|---|---|
| Bearer Token 鉴权 | OS 随机数生成 32 字节 hex token |
| 常量时间比较 | 使用 `subtle::ConstantTimeEq` 防止时序攻击 |
| 握手速率限制 | `/api/handshake` 端点 3 秒冷却，防暴力破解 |
| Auth Check 端点 | 扩展能区分"桌面端离线" vs "Token 不匹配" |

**密钥层**

| 措施 | 说明 |
|---|---|
| OS Keychain 存储 | API Key 存入 macOS Keychain / Windows Credential Manager，不进数据库 |
| Key Hint 机制 | 数据库仅存储密钥最后 4 位（用于 UI 展示），避免打开设置页时触发 Keychain 弹窗 |
| Ollama 支持 | 用户可以全部使用本地 Ollama 模型，AI 调用完全在本地完成，零数据外泄 |

#### AI 调用的隐私边界

这是用户最关心的灰色地带：AI 功能需要调用外部 API，那数据不是还是发出去了吗？

KnoYoo 的立场清晰：

- **AI 调用仅在用户主动使用时发生**（收藏触发清洗、手动重跑、搜索、对话）
- **调用的是用户自己配置的 API Key 和供应商**，KnoYoo 不做中间代理
- **用户可以选择完全本地化**：Ollama 支持所有 AI 功能，包括清洗、标签、搜索、对话
- 错误信息不泄露内部路径或数据库细节

#### 规划中的增强

| 功能 | 描述 | 状态 |
|---|---|---|
| 端到端加密多端同步 | 可选的跨设备同步，数据在传输和存储时全程加密，服务端不可读 | 规划中 |
| 扫描 PDF OCR | 纯扫描版 PDF 的本地 OCR 支持 | 规划中 |

## 四大支柱的关系

```
                    ┌──────────────────────────────┐
                    │    支柱四：本地 + 数据主权     │
                    │        （信任基础层）          │
                    └───────────────┬──────────────┘
                                    │
                    一切建立在"数据属于你"的信任之上
                                    │
           ┌────────────────────────┼────────────────────────┐
           │                        │                        │
  ┌────────▼─────────┐   ┌─────────▼─────────┐   ┌─────────▼─────────┐
  │  支柱一：无痛输入  │   │  支柱二：智能检索   │   │  支柱三：知识生长   │
  │  + 自动结构化     │──▶│   + AI 对话        │   │     可视化         │
  │    （入口）       │   │    （出口）         │   │    （驱动力）      │
  └──────────────────┘   └───────────────────┘   └───────────────────┘
         存                       用                       感
```

- **支柱四是地基**——没有信任，用户不会把知识托付给系统
- **支柱一是入口**——无痛输入降低门槛，自动结构化创造价值
- **支柱二是出口**——找回和复用是知识库存在的意义
- **支柱三是引擎**——成长可视化驱动长期使用，把工具变成习惯

四者形成完整闭环：

> **信任 → 存入 → 用起来 → 看到成长 → 更愿意存入 → ...**

## 用户画像

### 画像 1：信息焦虑的知识工作者（核心启动用户）

| 维度 | 描述 |
|---|---|
| **典型身份** | 25-35 岁，程序员 / 产品经理 / 研究生 / 独立开发者 |
| **日常行为** | 每天大量浏览技术博客、YouTube 教程、B站知识视频、Twitter/X |
| **核心痛点** | "我记得看过一篇关于 XXX 的文章，但死活找不到了"。收藏夹几百条从不回看，浏览器标签开 40 个舍不得关 |
| **心理特征** | 对知识有囤积欲望，但缺乏整理能力和时间。有轻微的"信息 FOMO"——怕错过好内容 |
| **为什么选 KnoYoo** | 不需要手动整理（AI 自动做）、不需要换工具链（浏览器扩展无缝接入）、数据在本地不担心服务关停 |
| **付费意愿** | 中等。愿意为"省时间"付费，但对订阅制敏感，更偏好买断或自带 API Key |
| **典型场景** | 写技术方案时搜自己的智库："我之前看过一个分布式锁的最佳实践，用的什么方案来着？" |

### 画像 2：隐私敏感的深度阅读者

| 维度 | 描述 |
|---|---|
| **典型身份** | 30-45 岁，律师 / 金融从业者 / 学术研究者 / 自由职业者 |
| **日常行为** | 阅读量大且内容敏感（行业报告、法律文书、竞品分析），收藏的内容不希望任何第三方看到 |
| **核心痛点** | 现有云端工具让他不安——数据在别人服务器上。但本地工具又太原始、缺乏智能能力 |
| **心理特征** | 对"数据主权"有强烈意识，可能用过自建 NAS 或加密邮箱。技术能力中上，愿意折腾但不想折腾太多 |
| **为什么选 KnoYoo** | **100% 本地存储**是决定性因素。可以用 Ollama 完全离线运行 AI，没有任何数据外泄风险 |
| **付费意愿** | 较高。为隐私和数据安全付费意愿强，但需要信任产品的开源透明度 |
| **典型场景** | 把竞品分析报告、客户资料、行业研究全部导入 KnoYoo，用 AI 对话快速检索关键信息 |

### 画像 3：终身学习型内容消费者（未来增长用户）

| 维度 | 描述 |
|---|---|
| **典型身份** | 22-30 岁，大学生 / 考研党 / 自学编程/设计/语言的学习者 |
| **日常行为** | 大量看 B站教程、YouTube 课程、电子书，但看完就忘，笔记散落在多个 App 里 |
| **核心痛点** | "我学了很多但感觉什么都没记住"。缺乏学习的正反馈和成就感 |
| **心理特征** | 需要**可视化的进步感**来维持学习动力。喜欢打卡、里程碑、进度条 |
| **为什么选 KnoYoo** | 视频字幕自动转文字让学习内容可搜索；书架的阅读分组给学习路径以结构；发现页让学习有回顾 |
| **付费意愿** | 低。学生群体价格敏感，但如果有免费 tier + 自带 API Key 模式可以接受 |
| **典型场景** | 看完一个 3 小时的 React 教程，全部字幕入库，3 个月后搜"useEffect cleanup"直接找到对应片段 |

### 优先级策略

**画像 1 是启动用户**——技术能力足够（能装桌面端 + 配 API Key）、痛点最强、社区传播力最大。先把这群人服务好。

**画像 2 是高价值用户**——付费意愿最强，对隐私特性的需求与产品基因完全吻合。

**画像 3 是增长引擎**——人群基数最大，但需要支柱三（知识生长可视化 + 正反馈）显著强化后才能有效触达。

## 与竞品的本质区别

| 竞品 | 它做什么 | KnoYoo 的区别 |
|---|---|---|
| **Pocket / Raindrop.io** | 收藏网页 | 只解决"存"，没有 AI 自动结构化，没有语义搜索和知识对话，数据在云端 |
| **Notion** | 笔记 + 知识库 | 强大但需要大量手动整理，输入成本高，数据在云端 |
| **Obsidian** | 本地 Markdown 笔记 | 本地优先但以"写"为核心，不解决"被动消费的信息如何自动入库"的问题 |
| **Evernote** | 全能笔记 | 云端存储隐私不可控，AI 能力弱，剪藏后仍需手动整理 |
| **ChatGPT / AI 助手** | AI 对话 | 没有持久化知识库，对话结束知识就丢了，无法与个人积累交互 |
| **DEVONthink** | 本地文档管理 | 本地优先但无 AI 自动结构化，学习曲线陡峭，仅 macOS，价格高 |
| **Readwise / Reader** | 阅读高亮同步 | 聚焦阅读场景，不覆盖视频/书籍/多源输入，部分功能依赖云端 |

**KnoYoo 的独特位置：本地优先 × AI 自动结构化 × 多源输入 × 知识对话 × 成长可视化。**

目前市场上没有产品同时占据这五个维度。
